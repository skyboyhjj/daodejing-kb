# 审核工作流优化方案

> **版本**：v1.0 (Draft)
> **日期**：2026-05-14
> **状态**：设计阶段，待确认后实施
> **涉及文件**：`admin/family-review.js`、`admin/family-review.html`、`admin/family-review.css`、`server.js`、`api/_shared/metadata-store.js`

---

## 一、流程合规性评估

### 1.1 用户提出的 6 步审核流程

```
步骤 1: 开始审核，发现需要修改
步骤 2: 在审核备注填写修改意见
步骤 3: 点击"修改"按钮
步骤 4: 状态从 reviewing → revision_needed
步骤 5: 系统根据修改意见重新生成元数据，并标注具体修改内容
步骤 6: 将章节重新提交到 reviewing 状态
```

### 1.2 现有状态机定义

`api/_shared/metadata-store.js:20-25`：

```
VALID_TRANSITIONS = {
    pending:           [reviewing],
    reviewing:         [approved, revision_needed, pending],
    revision_needed:   [reviewing],
    approved:          [revision_needed]
}
```

### 1.3 合规性判定

| 步骤 | 状态转换                    | 合法性         | 说明                                                                                                  |
| ---- | --------------------------- | -------------- | ----------------------------------------------------------------------------------------------------- |
| 1–4  | reviewing → revision_needed | ✅ 合法         | `reviewing` 的目标状态列表中包含 `revision_needed`                                                    |
| 6    | revision_needed → reviewing | ✅ 合法         | `revision_needed` 的唯一目标状态就是 `reviewing`                                                      |
| 5    | N/A（内容操作，非状态转换） | ✅ 不涉及状态机 | 步骤5 是元数据内容重新生成操作，发生在 `revision_needed` 状态下，不触发状态转换，因此不涉及状态机校验 |

**结论：用户提出的 6 步流程完全合规，无需修改状态机定义。**

### 1.4 关键差距：步骤5

步骤5（"系统根据修改意见重新生成元数据并标注修改内容"）在当前系统中**没有对应实现**。目前 `reviewing → revision_needed` 仅做两件事：

1. 在 `review_history` 中追加一条记录，携带自由文本 `notes`
2. 更新 `review_status` 字段

**不会触发**：基于修订意见的内容重新生成、字段级修改标注、before/after 内容快照。这是本次优化的核心填补区域。

## 二、现状分析

### 2.1 现有审核控制台架构

```
admin/family-review.html ─── SPA 界面
    ├── 顶部栏：4 种状态统计徽章
    ├── 左侧：可筛选/搜索的章节列表
    └── 右侧：详情面板
         ├── 操作按钮栏 (renderActionButtons)
         ├── 审核备注 textarea (通用自由文本)
         ├── 基本信息卡片 (core_idea/safety_notes/interaction_points/parent_tips)
         ├── 安全检查表 (Article 11, 11 项勾选)
         └── 审核历史时间线 (仅展示 action/by/at/notes)
```

API 端点 (`server.js`)：

| 端点                              | 方法   | 鉴权        | 功能                |
| --------------------------------- | ------ | ----------- | ------------------- |
| `/api/metadata/stats`             | GET    | 无          | 聚合统计            |
| `/api/metadata?chapter=N`         | GET    | 无          | 单章详情            |
| `/api/metadata?status=X&search=Y` | GET    | 无          | 列表（筛选+搜索）   |
| `/api/metadata`                   | PUT    | ADMIN_TOKEN | 状态转换 / 字段更新 |
| `/api/metadata?chapter=N`         | DELETE | ADMIN_TOKEN | 删除章节            |

### 2.2 当前存在的问题

| 领域         | 问题                                              | 影响                                         |
| ------------ | ------------------------------------------------- | -------------------------------------------- |
| **修改意见** | 单一自由文本 textarea，无结构化字段区分           | 无法精准定位哪些字段需要修改、修改方向是什么 |
| **修改意见** | 操作完成后 textarea 被清空 (`notesEl.value = ''`) | 修改意见丢失，后续无法回顾                   |
| **内容对比** | 完全没有 before/after 对比能力                    | 审核者无法看到修订前后的差异                 |
| **状态转换** | 使用浏览器原生 `confirm()`                        | 风格不统一、无操作预览、无 loading 反馈      |
| **审核历史** | 仅记录 action/by/at/notes                         | 无法追溯内容快照、无法关联修订意见与后续修改 |
| **整体 UX**  | 操作反馈弱，按钮状态管理简单                      | 审核体验不够流畅                             |

- 重新打开章节时自动回显上次的修订意见

### 4.2 新增数据结构：`revisions`

## 五、优化领域 2：前后内容对比视图

### 5.1 设计目标

- 在 `revision_needed → reviewing` 后，审核者能直观看到修改前后的差异
- 支持逐字段对比（core_idea 文本 / safety_notes 数组 / interaction_points 数组）
- 差异用颜色高亮标注

### 5.2 新增数据结构：`content_snapshot`

```javascript
// review_history 条目的新增字段（optional，旧条目兼容）
content_snapshot: {
    core_idea: "真正懂的人不炫耀...",
    safety_notes: [
        "避免将'不言'等同于不说话或沉默是金...",
        "避免用'锋芒'、'尖锐'等词批评孩子...",
        "避免将'和其光，同其尘'解释为失去自我..."
    ],
    interaction_points: [
        { topic: "不炫耀的力量", age_4_6: "...", age_7_9: "...", age_10_12: "..." },
        { topic: "特别还是融入", age_4_6: "...", age_7_9: "...", age_10_12: "..." }
    ],
    parent_tips: "和孩子一起安静地坐一会儿吧..."
}
```

### 5.3 快照保存时机

| 触发条件                    | 保存内容         | 用途                          |
| --------------------------- | ---------------- | ----------------------------- |
| reviewing → revision_needed | 当前所有字段值   | 保存"修改前"的样子            |
| revision_needed → reviewing | 变更后所有字段值 | 保存"修改后"的样子，用于 diff |

### 5.4 Diff 算法策略

| 字段类型                              | 差异算法                                                                        |
| ------------------------------------- | ------------------------------------------------------------------------------- |
| 文本字段 (`core_idea`, `parent_tips`) | 按中文句号 `。` 拆分句子，逐句 LCS (Longest Common Subsequence) 匹配            |
| 数组字段 (`safety_notes`)             | 逐条文本相似度匹配（Levenshtein distance > 0.6 视为同一项），标记新增/删除/修改 |
| 对象数组 (`interaction_points`)       | 按 `topic` 字段匹配，相同 topic 内的 age 字段逐项对比                           |

### 6.4 操作按钮状态管理

```javascript
// 状态: idle → loading → success/error → idle

function performAction(chapterNum, action) {
    // 1. 禁用所有操作按钮，显示 loading
    setButtonsDisabled(true);
    showButtonSpinner(action);
    
    // 2. API 调用
    http('PUT', '/api/metadata', payload)
        .then(handleSuccess)
        .catch(handleError)
        .finally(function() {
            // 3. 恢复按钮（延迟，给用户看清反馈的时间）
            setTimeout(function() {
                setButtonsDisabled(false);
                hideButtonSpinner();
            }, 800);
        });
}
```

## 七、优化领域 4：审核历史完整性

### 7.1 设计目标

- 审核历史不仅是操作日志，更是**完整审核轨迹**
- 每个关键节点保存内容快照，支持事后追溯
- 修订意见与后续修改之间建立可追溯关联

### 7.2 增强后的 history 条目结构
