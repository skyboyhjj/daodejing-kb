# 审核工作流详细说明

## 状态机

```
┌──────────┐  开始审核(→reviewing)  ┌───────────┐
│  pending  │───────────────────────→│ reviewing  │
│  (待审核)  │                       │  (审核中)   │
└──────────┘                        └─────┬──────┘
     ↑                                   │
     │  退回待审(→pending)          ┌────┴─────┐
     │  (仅当审核员一致时)            │          │
     │                         审核通过    需要修改
     │                         →approved  →revision_needed
     │                             │          │
     │                        ┌────▼──┐ ┌────▼──────────┐
     └────────────────────────│approved│ │revision_needed│
                 改为需修改    │ (已通过) │ │  (需修改)      │
                 ←────────────└────────┘ └───────────────┘
                                           │
                                           │ 重新审核(→reviewing)
                                           └──────────→ reviewing
```

## 状态说明

| 状态              | 含义                | 可执行操作                                                                    |
| ----------------- | ------------------- | ----------------------------------------------------------------------------- |
| `pending`         | AI 生成，待人工审核 | 开始审核 → `reviewing`                                                        |
| `reviewing`       | 审核进行中          | 审核通过 → `approved`<br>需要修改 → `revision_needed`<br>退回待审 → `pending` |
| `approved`        | 审核通过，可上线    | 改为需修改 → `revision_needed`                                                |
| `revision_needed` | 需修改后重审        | 重新审核 → `reviewing`                                                        |

## 软锁定机制

- 开始审核时，`reviewed_by` 记录当前操作者标识
- 其他审核员看到已锁定的章节时会收到提示
- 锁定状态在状态变更为 `approved` 或 `pending` 时清除

## 审核记录

每次状态变更自动生成一条 `review_history` 记录：

```json
{
  "action": "reviewing",
  "by": "admin",
  "at": "2026-05-12T05:36:37.490Z",
  "notes": "开始逐项检查"
}
```

历史记录按时间倒序显示在审核控制台的时间线中。

## 安全审核检查表使用

1. 打开章节详情后，逐项阅读内容
2. 对每个检查项，确认内容符合要求后勾选
3. **审核通过前必须 11 项全部勾选**（控制台会在未全选时弹出确认框）
4. 如某项不符合，应退回 `revision_needed` 状态并在备注中说明

### 检查项清单（11 项）

| # | 检查项 | 检查标准 |
|---|--------|---------|
| 1 | 无死亡/灾难/暴力 | 不含战争、死亡、灾难、暴力等可能引发儿童恐惧的表述 |
| 2 | 无成人主题 | 不含性、暴力、政治敏感等不适宜儿童的内容 |
| 3 | 无成瘾性设计 | 语言不诱导沉迷、不制造焦虑依赖 |
| 4 | 语言难度匹配 | 表述符合 4-12 岁儿童的认知水平 |
| 5 | 无负面引导 | 不鼓励攀比、撒谎、自私、攻击等行为 |
| 6 | 无身体焦虑 | 不涉及身材/容貌评价或比较 |
| 7 | 无道德绑架 | 不使用评判式、威吓式语言 |
| 8 | 语气温暖 | 尊重儿童主体性，语气亲和、鼓励而非说教 |
| 9 | 无道德误读风险 | 不将《道德经》概念（无为、返璞归真、不争等）与不守规则、不讲礼貌等不良行为相关联 |
| 10 | 不对孩子行为做评判 | 对话模板中不对孩子的行为做道德评判，保持客观描述而非价值判断 |
| 11 | 正确的不良行为回应策略 | 当孩子描述犯错或不听话时，回应策略是平常心回应情绪，不借机说教或美化 |

## 删除策略

- `approved` 状态的章节不可删除（需要在控制台 UI 中隐藏删除按钮，API 层面也会拒绝）
- 其他状态的章节可删除，但会弹出二次确认
- 删除操作为硬删除（从 JSON 中移除），但 `data/family_metadata_v1.0_backup.json` 保留了初始备份

## 与 extract_meta.py 的集成

`extract_meta.py` 生成元数据后：
1. 自动设置 `review_status: "pending"`
2. 生成 `review_history` 首条记录（`action: "created"`）
3. 不会覆盖已通过审核的内容（检查 `review_status` 是否为 `approved`）
