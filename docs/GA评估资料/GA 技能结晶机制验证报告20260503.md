## 二、测试方法

### 2.1 自动化测试流程

为避免人工交互带来的不确定性，我们编写了自动化测试脚本 `memory/verify_ga_crystallization.py`，采用“快照-执行-比对”三段式验证：

1. **快照阶段**：记录 `memory/` 目录下所有文件的完整列表和 `global_mem_insight.txt` 的行数与内容
2. **执行阶段**：通过 GA 命令行模式提交任务，等待进程结束
3. **比对阶段**：对比执行前后的文件集合，识别新增文件（特别是 `*_sop.md`），分析 L1 索引变化

### 2.2 测试用例设计

为了验证结晶机制的边界，我们设计了五个递进式测试用例：

| 用例 | 任务特征 | 预期结晶 | 设计理由 |
|:---|:---|:---|:---|
| `crystal_test_01` | 单步计算任务 | 否 | 验证“过于简单则不需要记忆” |
| `crystal_test_02` | 多工具调用 + 创建可执行脚本 | **可能触发** | 任务指令中明确要求“记录下来”，创建了有复用价值的 Python 脚本 |
| `crystal_test_03` | 文件搜索（与已有能力高度重复） | 否 | 验证“新颖性”在结晶决策中的作用 |
| `crystal_test_04` | 外部信息获取（web_scan）+ 多工具编排 | 可能触发 | 有外部信息获取和明确的跨会话复用价值 |
| `crystal_test_05` | 包含错误恢复分支的复合任务 | 可能触发 | 验证“踩坑后学会”是否触发结晶 |

**L1 索引更新**：`global_mem_insight.txt` 新增 `check_files_sop` 关键词索引。

### 3.4 未触发结晶的用例分析

**`crystal_test_04`**（访问 Sophub 网站获取信息并保存）和 **`crystal_test_05`**（错误恢复 + 网站连通性测试）虽然涉及多工具调用，且我们预期可能触发结晶，但 GA 未生成新的 SOP 文件。

可能原因：
1. LLM 判断这些任务的执行路径不具备“跨会话复用”的必要性
2. 任务执行中存在依赖安装等操作，退出码为 1，可能被判定为不够“干净”的成功
3. 结晶由 LLM 完全自主决定，不受外部规则约束

## 五、关键发现

### 5.1 核心结论：结晶是 LLM 的涌现行为，而非系统级的确定性机制

本次测试最核心的发现是：**GA 的技能结晶并非由硬编码规则触发，而是一个完全由 LLM 自主判断的涌现过程。** 即使任务成功完成，如果 LLM 在内部推理中不认为该经验“值得长期记忆”，就不会调用记忆更新工具，也就不会产生结晶。

### 5.2 对慧惠项目的指导意义

慧惠项目的目标是构建一个“会生长、会修剪、会遗忘的数字生命”，其技能管理机制（我们称之为“五行流转引擎”）需要明确区分以下职责：

| 职责 | GA 现有机制 | 慧惠增强方向 |
|:---|:---|:---|
| **木·生（技能生长）** | LLM 自主判断 | 增加**显式质量门控**：结晶化评分 ≥0.6 才通过 |
| **金·克（技能修剪）** | 无 | 实现 `MetalRestrainer`：健康度评估 + 自动归档 |
| **火·化（技能整合）** | 无 | 实现 `FireTransformer`：相似技能合并 |
| **产物质量校验** | 依赖 LLM 自我约束 | 增加**程序化校验**：易变状态检测、行动引用检测 |

我们不是在替换 GA 的结晶机制，而是为其补充一套**外部的、可观测的、可调控的**技能生命周期管理系统。
