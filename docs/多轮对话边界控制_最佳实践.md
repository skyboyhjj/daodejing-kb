# 多轮对话边界控制 · 最佳实践

> **版本**：v1.0  
> **日期**：2026-05-18  
> **适用范围**：基于无状态 LLM API 的多轮对话场景  
> **已验证场景**：慧惠晚间回顾（DeepSeek）、亲子共读对话（`family.js`）  
> **关联 Skill**：`ai-intent-router`（前缀注入路由）

---

## 一、问题定义

### 1.1 核心矛盾

基于 HTTP 的 LLM API（如 DeepSeek、OpenAI）是**无状态**的：每次请求独立，不保留前一轮对话的"进度记忆"。但很多交互场景**天然是多轮的**——需要在多个请求-响应周期中维持一个连贯的流程。

当 System Prompt 描述了一个"完整流程"（如"先做 A，再做 B，最后做 C"），LLM 在**单次请求**中可能**一次性输出全部步骤**，跳过等待用户输入的环节。

### 1.2 典型症状

| 症状 | 表现 | 根本原因 |
|:---|:---|:---|
| **跳过等待** | AI 在第一轮就输出了完整小结，而小结本应在用户回答后才生成 | System Prompt 将步骤 1-3 描述为连续指令块，LLM 没有收到"在步骤 1 停止"的显式指令 |
| **格式泄露** | AI 的输出中包含了"本应在下一轮才显示的格式模板" | LLM 看到了整个流程描述，倾向于"一次性完成任务" |
| **上下文污染** | 后续轮次中 AI 重复执行了前面的步骤 | LLM 在每轮都重新评估所有 System Prompt 指令，没有"已执行"的标记 |

---

## 二、解决方案：显式轮次边界

### 2.1 核心原则

**在 System Prompt 中用显式的、不可误解的边界指令，将多轮流程切分为独立的"本轮任务"。每个边界包含三个要素：**

1. **做什么** — 本轮仅执行的具体操作
2. **不做什么** — 明确禁止的操作（防止"溢出"）
3. **停止信号** — 显式的"本轮到此结束"指令

### 2.2 模板

```
【本轮回复——只做这一件事】
{本轮的单一任务描述}
——本轮到此结束。不输出{后续内容}、不输出{禁止内容}。等待用户回答。

【用户回答后——再按以下格式生成{下一轮内容}】
{下一轮的格式和约束}
```

### 2.3 在本项目中的实际代码

**文件**：`api/_shared/system-prompt.js` 第 67-83 行

```javascript
'当用户消息以 [MIRROR:EVENING] 开头时，启动晚间回顾流程（分两轮对话，不可在单轮中全部完成）：\n' +
'\n' +
'【本轮回复——只做这一件事】\n' +
'从对话历史中统计今日完成的镜鉴事件数量（识别四维评分完成的对话），不要编造数字。\n' +
'然后只输出这一句话：\n' +
'"今天你完成了{n}次镜鉴。有没有哪一刻，你感觉自己最接近\'道\'？"\n' +
'——本轮到此结束。不输出小结、不输出格式、不输出任何后续内容。等待用户回答。\n' +
'\n' +
'【用户回答后——再按以下格式生成今日小结】\n' +
'\n' +
'🌿 今天辛苦了。这是你的镜鉴小结。\n' +
'...'
```

### 2.4 关键要素拆解

| 要素 | 实际文本 | 作用 |
|:---|:---|:---|
| **轮次限定** | "分两轮对话，不可在单轮中全部完成" | 在最外层声明轮次数，建立全局预期 |
| **任务边界** | "【本轮回复——只做这一件事】" | 用醒目的视觉标记（【】）创建不可忽视的边界 |
| **单一任务** | "从对话历史中统计...然后只输出这一句话" | 精确描述本轮唯一要做的事 |
| **输出限制** | "只输出这一句话" | 明确禁止额外输出 |
| **停止指令** | "——本轮到此结束。" | 显式告诉 LLM"你的回复在这里停止" |
| **禁止溢出** | "不输出小结、不输出格式、不输出任何后续内容" | 三重否定，覆盖所有可能的"溢出"行为 |
| **等待信号** | "等待用户回答" | 告诉 LLM 下一轮的条件 |
| **下轮锚点** | "【用户回答后——再按以下格式生成今日小结】" | 用相同的边界标记链接到下一轮 |

## 三、两种实现路径

### 3.1 路径 A：System Prompt 边界控制（LLM 端）

**适用场景**：后端的 AI 对话，每轮通过独立的 HTTP 请求完成。

**实现方式**：在 System Prompt 中写入轮次边界指令（如上节所示），LLM 在每次请求时遵循约束。

**优势**：
- 无需任何状态管理
- 不修改前端代码
- LLM 自动控制节奏

**劣势**：
- 依赖 LLM 的指令遵循能力（非 100% 可靠）
- 需要精心设计 Prompt（三重要素缺一不可）
- token 消耗略高（边界指令占一定篇幅）

**代码参考**：`api/chat.js`（通过 `buildSystemPrompt()` 控制）、`api/_shared/system-prompt.js`

### 3.2 路径 B：前端轮次计数器（Client 端）

**适用场景**：前端驱动的对话，轮次通过 JS 变量追踪。

**实现方式**：在前端维护 `roundIndex` 变量，每轮 +1。根据 `roundIndex` 和 `maxRounds` 决定是否继续、切换到哪个阶段。

**优势**：
- 100% 可靠（不受 LLM 非确定性影响）
- 可实现复杂的流程分支
- 无需消耗 LLM token

**劣势**：
- 需要前端状态管理
- 后端 AI API 仍然是独立调用，前端需管理对话上下文

**代码参考**：`js/family.js`

```javascript
// family.js 中的轮次管理
var state = {
    roundIndex: 0,      // 当前章节内的轮次
    maxRounds: 10,      // 每章最多 10 轮
    totalRounds: 0,     // 全对话累计轮次
    stage: 'intro'      // 当前阶段：intro | dialogue | closing
};

// 每轮发送后递增
function sendMessage() {
    // ...
    if (state.totalRounds >= state.maxRounds) {
        endChapter();   // 达到上限，主动结束
        return;
    }
    sendNextRound();    // 继续下一轮
}
```

### 3.3 路径对比

| 维度 | 路径 A（LLM 边界） | 路径 B（前端计数器） |
|:---|:---|:---|
| **可靠性** | 高（需精心设计 Prompt） | 100% |
| **灵活性** | 低（依赖 LLM 理解） | 高（任意 JS 逻辑） |
| **状态管理** | 零（无状态） | 需要（`roundIndex` 等变量） |
| **适用场景** | 简单、线性流程 | 复杂、有分支的流程 |
| **本项目案例** | 晚间回顾（2 轮固定） | 亲子共读（≤10 轮动态） |

## 四、最佳实践建议

### 4.1 设计轮次边界时的检查清单

- [ ] 是否在最外层声明了总轮次数？— "分 N 轮对话，不可在单轮中全部完成"
- [ ] 每轮是否有独特的视觉标记？— `【本轮回复】` / `【用户回答后】`
- [ ] 每轮是否只描述一个任务？— 不将多个步骤放在同一轮
- [ ] 是否明确禁止了"溢出"内容？— "不输出任何后续内容"
- [ ] 是否有显式的停止指令？— "本轮到此结束"
- [ ] 是否有"等待用户"的触发条件？— "等待用户回答"
- [ ] 下一轮的锚点是否清晰？— "用户回答后——再按以下格式..."
- [ ] 是否考虑了"用户可能跳过"的兜底？— "用户可随时回复'不用'跳过"

### 4.2 Prompt 设计原则

**原则 1：用视觉标记创建不可忽视的边界**

```
✅ 好的：    【本轮回复——只做这一件事】
❌ 差的：    首先，你需要做 A。然后等待。再然后做 B。
```

**原则 2：每轮任务描述必须"唯一且排他"**

```
✅ 好的：    只输出这一句话："今天你完成了{n}次镜鉴。有没有哪一刻..."
❌ 差的：    先统计事件数量，然后输出一句话，接下来等待。
```

"接下来"这个词暗示 LLM 还有后续，容易触发"继续输出"。

**原则 3：用三重否定防止溢出**

```
✅ 好的：    不输出小结、不输出格式、不输出任何后续内容。
❌ 差的：    不要输出太多内容。
```

"不要输出太多"过于模糊，LLM 可能理解为"可以输出一点"。

**原则 4：边界标记在全文保持一致**

```
✅ 好的：
    【本轮回复——只做这一件事】      ← 注意：使用了【】
    【用户回答后——再按以下格式...】  ← 一致使用【】

❌ 差的：
    **本轮任务**                     ← markdown bold
    ── 用户回答后 ──                ← 用横线
```

### 4.3 常见的 Prompt 反模式

| 反模式 | 问题 | 修正 |
|:---|:---|:---|
| "接下来..." | 暗示 LLM 有后续内容 → 触发连续输出 | 改为"本轮只输出..." |
| "你可以..." | 给 LLM 选择权 → 可能选择"提前输出后续" | 改为"你必须只..." |
| 在末尾描述下一轮 | LLM 看到"下一轮"就想"现在做完" | 把下一轮放在独立的边界块中 |
| "等待"而非"停止" | "等待"暗示 LLM 还在"进行中" | 必须说"本轮到此结束" |
| 自然语言流程描述 | 如"先统计，然后提问" → LLM 可能一次性完成 | 用【】块显式分隔 |

### 4.4 测试验证方法

**方法 1：空对话测试**

向 LLM 发送仅含系统提示和单条触发消息的请求（无历史对话），检查输出是否仅包含第一轮内容。

```bash
curl -X POST {API_URL} \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [
      {"role": "user", "content": "[MIRROR:EVENING] 晚间回顾"}
    ]
  }'
# 期望：仅输出第一轮内容（一句话），不包含小结格式
```

**方法 2：两轮完整性测试**

模拟完整两轮对话，检查第二轮是否正确识别"用户已回答"并生成完整内容。

```javascript
// Step 1: 发送触发消息 → 得到第一轮回复
// Step 2: 将 step1 的回复作为 assistant 消息，追加用户回答
// Step 3: 发送 → 检查第二轮是否生成完整的后续内容
```

**方法 3：边界压力测试**

连续发送 3 次同样的触发消息，检查 AI 是否每次都从第一轮重新开始（而非"记住"之前的进度）。

**方法 4：跳过测试**

在第二轮时回复"不用"，检查 AI 是否正确停止流程。

## 六、验收标准

- [ ] 触发多轮流程后，AI 第一轮仅输出当前轮次的内容（不泄露后续格式）
- [ ] 用户回答后，AI 正确进入下一轮（不重复第一轮）
- [ ] 用户回复"不用"后，AI 正确停止流程
- [ ] 连续触发 3 次，每次 AI 都从第一轮重新开始
- [ ] System Prompt 使用了显式边界标记（【】）和三重否定
- [ ] 每轮任务描述"唯一且排他"
