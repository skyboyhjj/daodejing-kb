## 二、元数据提取脚本设计

### 2.1 脚本定位

不依赖人工逐章编写，而是利用 LLM **辅助提取**元数据初稿，人工只需审核修改，大幅降低人力成本。

### 2.2 操作文件

| 文件 | 操作 | 说明 |
|:---|:---|:---|
| `scripts/extract_meta.py` | **新建** | 元数据提取脚本 |
| `data/family_metadata.json` | **扩展** | 在现有 3 章基础上追加新章节元数据 |
| `data/family_metadata_review.md` | **扩展** | 审核追踪表，增加新章节的审核记录 |

### 2.3 提取流程

1. **读取章节源文件**：`daodejing-kb/chapters/ch{XX}.html`
2. **提取文本内容**：去除 HTML 标签，提取所有 `data-depth="l1"`（白话版）文本
3. **LLM 提取元数据**：将白话版文本发送至 DeepSeek API，使用专门的元数据提取 Prompt
4. **格式化为标准 JSON**：与现有 `family_metadata.json` 保持一致
5. **输出到文件**：追加到 `data/family_metadata_raw.json`（未审核，待人工确认）

### 2.4 元数据提取 Prompt

```
你是一个《道德经》亲子教育内容编辑。请从以下章节解读中，提取适合亲子共读的核心信息。

【章节原文与白话解读】

{chapter_content}

【提取要求】
1. **core_idea**（核心观点）：用一句话概括这一章最适合讲给孩子听的核心思想。要简单、有画面感，孩子能听懂。不超过 30 字。
2. **interaction_points**（互动话题）：提取 2 个适合亲子互动的讨论点。每个话题包含：
   - topic：话题名称（如 "水的厉害之处"）
   - age_4_6：针对 4-6 岁孩子的引导方向（用大自然、小动物、日常生活比喻）
   - age_7_9：针对 7-9 岁孩子的引导方向（用故事、角色扮演、简单类比）
   - age_10_12：针对 10-12 岁孩子的引导方向（引入更多原文、探讨更抽象的概念）
   若某年龄段不适合此话题，该字段设为 null。
3. **parent_tips**（给家长的话）：一段温暖的话，告诉家长如何和孩子聊这一章。不超过 80 字。
4. **safety_notes**（安全提示）：列出这一章在亲子对话中需要避免的话题或表达方式。如无特别需要注意的，设为空数组 []。

【输出格式】
严格按以下 JSON 格式输出，不要输出其他任何内容：
{
  "chapter": {chapter_number},
  "title": "{chapter_title}",
  "core_idea": "...",
  "safety_notes": ["...", "..."],
  "interaction_points": [
    {
      "topic": "...",
      "age_4_6": "...",
      "age_7_9": "...",
      "age_10_12": "..."
    }
  ],
  "parent_tips": "..."
}
```

### 2.5 章节源文件读取规则

- 源文件路径：`public/chapters/ch{XX}.html`（XX 为零填充两位数，如 ch01.html）
- 提取内容：所有 `data-depth="l1"` 标签内的文本（白话版解读）
- 如果某章的白话版内容为空或不存在，跳过该章，记录到日志中

### 2.6 执行方式

脚本支持单个章节测试和批量提取两种模式。批量提取时自动跳过已存在的章节（避免重复），支持断点续跑。每章提取耗时约3-5秒（含API调用），全量81章约5-7分钟。

## 四、审核流程

### 4.1 审核清单

每章元数据审核需逐项确认以下内容：

| 审核项 | 标准 | 不通过示例 |
|:---|:---|:---|
| **内容安全** | 无儿童恐惧话题（死亡、灾难、暴力） | “战争”、“洪水”等词汇 |
| **年龄适配** | 各年龄段的引导方向与该年龄段认知水平匹配 | 4-6 岁出现抽象哲学概念 |
| **哲学准确性** | 核心观点不偏离该章原意 | 曲解“无为”为“什么都不做” |
| **亲子可操作性** | 给家长的话具体、可立刻执行 | “和孩子聊聊哲学”等空泛建议 |
| **话题区分度** | 两个互动话题有明显差异 | 两个话题实质上问的是同一件事 |

### 4.2 审核状态管理

元数据初稿自动设定为 `review_status: "pending"`，审核通过后修改为 `"approved"` 并附上审核人信息，只有已审核通过的章节才会被缓存预热脚本处理。


## 五、缓存预热脚本

### 5.1 脚本定位

审核通过后，批量调用 API 生成前 3 轮对话并缓存，确保用户首访即流畅体验。

### 5.2 操作文件

| 文件 | 操作 | 说明 |
|:---|:---|:---|
| `scripts/warmup_cache.py` | **新建** | 缓存预热脚本 |

### 5.3 预热逻辑

遍历已审核章节，为每个年龄段的每一轮对话生成LLM响应并写入缓存文件，每轮间隔1秒以遵守API频率限制，失败自动重试最多3次，重试仍失败的章节记录到日志供人工排查。
