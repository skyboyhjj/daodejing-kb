## 二、审核流程

### 2.1 状态机（`api/_shared/metadata-store.js:20-25`）

```
     pending ──────→ reviewing ──────→ approved
                       │  ↑                 │
                       │  ├── revision_needed│
                       └──┤                  │
                          └──────────────────┘
     (reviewing 也可退回 pending)
```

| 当前状态 | 允许的目标状态 | 说明 |
|----------|:-------------:|------|
| `pending` | `reviewing` | 开始审核 |
| `reviewing` | `approved`, `revision_needed`, `pending` | 通过/需修改/退回 |
| `revision_needed` | `reviewing` | 重新审核 |
| `approved` | `revision_needed` | 紧急撤回（需二次确认） |

### 2.2 审核控制台（`admin/family-review.html` + `admin/family-review.js`）

**界面三段式布局：**
- **顶栏**：实时统计数（pending / reviewing / approved / revision_needed）
- **左侧**：章节列表（支持搜索、状态筛选、键盘 j/k 导航）
- **右侧**：详情面板（元数据展示 + **Article 11 检查表** + 审核历史）

**操作流程以第25章为例：**

1. 管理员打开 `http://localhost:8080/admin/family-review.html`
2. 输入 Bearer Token（`ADMIN_TOKEN`，保存到 `sessionStorage`）
3. 左侧找到第25章（状态 = `pending`），点击「▶ 开始审核」
4. 后端验证 `pending → reviewing` 转换合法 ✓

6. 右侧显示：核心观点、安全注意事项、互动点、**11项安全检查表**
7. 逐项勾选检查表（第25章无暴力/无成人主题/无道德误读风险...）
8. 点击「✅ 审核通过」→ API `PUT /api/metadata`，`review_status: "approved"`
9. 后端：`reviewed_by` 清空、`reviewed_at` 设为当日、追加 `review_history`

## 三、缓存预热机制

### 3.1 双层缓存架构

| 层级 | 位置 | 容量 | TTL | 用途 |
|------|------|------|-----|------|
| **L1 内存缓存** | `server.js` 的 `_familyCache` Map | 进程生命周期 | 24小时 | 用户实时对话去重 |
| **L2 文件缓存** | `data/family_chat_cache.json` | 持久化（70KB） | 手动更新 | 预热批量预生成 |

### 3.2 缓存键生成（`server.js:182-207`）

```javascript
function familyCacheKey(chapter, ageGroup, history) {
    // chapter + ageGroup + 对话历史的 32-bit hash
    var historyStr = JSON.stringify(history);
    var hash = 0;
    for (var i = 0; i < historyStr.length; i++) {
        var c = historyStr.charCodeAt(i);
        hash = ((hash << 5) - hash) + c;
        hash = hash & hash; // 32-bit
    }
    return 'fc_' + chapter + '_' + ageGroup + '_' + hash;
}
```

**第8章 · 上善若水（age_7_9）缓存键示例：**

| 轮次 | 对话历史 | 缓存键 |
|------|----------|--------|
| 第1轮（开场白） | `[]`（空） | `fc_8_age_7_9_0` |
| 第2轮 | `[{huihui: "开场白..."}, {user: "我觉得水很厉害..."}]` | `fc_8_age_7_9_<hash1>` |
| 第3轮 | `[...4条消息]` | `fc_8_age_7_9_<hash2>` |

### 4.2 关键安全守卫

**API 级别：** `family_chat` handler 第550行
```javascript
if (chapterMeta.review_status !== 'approved') {
    res.writeHead(403, {...});
    res.end(JSON.stringify({ error: '第 ' + chapter + ' 章的元数据尚未通过审核' }));
    return;
}
```

**这意味着：** 只要 `review_status ≠ "approved"`，即使前端请求了该章，API 也会返回 403 禁止访问，DeepSeek 调用不会发生。这是**生产环境安全阀**。

### 4.3 元数据缓存 TTL

`metadata-store.js` 中设置了 **5 秒的 `_metadataCache` TTL**。这意味着：
- 审核通过后，**最多 5 秒**内所有后续 API 请求都会看到最新状态
- 在 Vercel/Cloudflare serverless 环境中，每次冷启动都会重新读取文件，天然实时

### 5.2 年龄适配机制的三层差别

### 5.3 浏览器端数据存储

**`sessionStorage`：**
- `admin_token`：管理员 Bearer Token（审核控制台使用）

**不持久化的数据（页面刷新后丢失）：**
- `state` 对象（stage, age, chapter, chaptersDone, conversationHistory...）
- 对话气泡 DOM 内容

**浏览器端缓存（HTTP 层）：**
- `family_metadata.json`：前端通过 XHR `GET /data/family_metadata.json` 获取 `parent_tips`（用于回退展示）
- 静态资源：`family.html`、`family.js`、`family.css` 的浏览器缓存遵循标准 HTTP 缓存策略

**第二天继续学习的处理：**

由于前端**不使用 localStorage/cookie 持久化对话状态**，第二天用户重新打开 `family.html` 后会：
1. 重新进入 `showWelcome()` → 年龄选择
2. 默认从第8章重新开始
3. **不会**从上次中断处继续（因无状态持久化）
4. 元数据 `parent_tips` 每次重新从服务器获取
